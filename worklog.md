## 2026-07-10 15:32 — вместе с Claude Code

Прогнали автоматизированный golden-path тест на живом деплое
(headless Chromium через Playwright, скрипт временный, не в репо):
signup → валидация пустой формы → успешная регистрация → неверный
пароль отклонён → успешный логин → dashboard (приветствие/статистика/
часы) → persistence после reload → clients (поиск/фильтр по статусу) →
добавление клиента (POST) → детальная модалка → заметка → напоминание
→ удаление клиента (DELETE, с подтверждением) → profile (данные,
редактирование имени, смена пароля со старым/новым паролем) → theme
toggle → logout → auth guard блокирует dashboard.html без сессии →
старый пароль после смены не работает, новый работает.

Результат: **27 из 27 проверок пройдены**, багов в приложении не
найдено. Единственная console-ошибка в браузере — `404` на
DELETE-запросе к `dummyjson.com` для клиента, добавленного через
POST — это ожидаемое поведение, явно описанное в PRD (P4.5): API не
сохраняет запись на сервере, поэтому не может её найти при удалении;
приложение корректно удаляет клиента из state локально в любом
случае.

Следующий шаг: постраничный разбор кода вместе, чтобы уверенно
объяснить любую часть на устном экзамене.

## 2026-07-14 15:37 — вместе с Claude Code

Я сам заметил `innerHTML` в `clients.js` и спросил про безопасность
(XSS). Разобрали: весь пользовательский текст (имя, компания, email,
текст заметки, URL картинки) уже был обёрнут в `escapeHtml()` — дыры
не было; `status`/`id` вставлялись без экранирования, но это не
уязвимость, так как оба поля всегда из закрытого набора значений
(select с 4 вариантами / числовой id), а не свободный текст. PRD
(P4.3) прямо разрешает оба варианта — `createElement` или
`insertAdjacentHTML`.

В качестве демонстрации попросил переписать `renderClients()` в
`js/clients.js` на `createElement`/`textContent` вместо
HTML-шаблонных строк — так безопасность обеспечена самой конструкцией
кода, экранирование вообще не нужно. Добавили `createEmptyState()` и
`createClientCard()`, `container.innerHTML = ...` заменили на
`container.replaceChildren(...)`. Проверили: `node --check` (синтаксис
ок), затем полный 27-шаговый Playwright-сценарий локально (написал
свой минимальный статический сервер вместо `npx serve`, потому что
`serve` по умолчанию делает 301-редирект `dashboard.html` → `/dashboard`,
из-за чего тест падал — это artefact локального сервера, не баг
приложения и не связано с этим изменением). Все 27 проверок прошли,
включая весь блок Clients (список, поиск, фильтр, добавление, заметки,
напоминание, удаление). Задеплоено не было ещё — изменение только
закоммичено, деплой на Vercel подхватит его автоматически при пуше.
